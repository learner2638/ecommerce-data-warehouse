# E-commerce Offline Data Warehouse
Dataset generated by:
https://github.com/learner2638/ecommerce-data-simulator
Offline Data Warehouse + Synthetic Data Generator + BI Dashboard

Dataset Scale: **10M Orders | 25M Order Items**

This project implements a complete **offline e-commerce analytics pipeline**, including:

- Large-scale synthetic data generation
- Layered data warehouse modeling
- Business metric aggregation
- Interactive BI dashboard visualization

The project simulates a simplified **data engineering analytics platform**.

---

# Project Architecture

The system architecture consists of two main components:

1. **E-commerce Data Simulator**
2. **Offline Data Warehouse**

```
Synthetic Data Generator
        ↓
Raw Transaction Data
        ↓
ODS Layer (Raw Data)
        ↓
DWD Layer (Detail Fact Tables)
        ↓
DWS Layer (Aggregated Metrics)
        ↓
ADS Layer (Business Analytics)
        ↓
Python BI Dashboard
```

This architecture follows the common **offline data warehouse pattern used in industry systems**.

```

user_cnt = 120_000
shop_cnt = 8_000
sku_cnt = 35_000
```

# Data Warehouse Design

The warehouse follows a **four-layer modeling strategy**:

```
ODS → DWD → DWS → ADS
```

---

## ODS Layer

Raw transactional data ingestion layer.

Example tables:

```
ods_orders
ods_order_items
ods_user_dim
ods_shop_dim
ods_sku_dim
```

---

## DWD Layer

Cleaned and standardized detail-level fact tables.

Example tables:

```
dwd_trade_order
dwd_trade_order_detail
dwd_trade_refund_detail
```

---

## DWS Layer

Aggregated service layer providing analytical metrics.

Example tables:

```
dws_trade_day_summary
dws_trade_category_day_summary
dws_trade_shop_day_summary
```

---

## ADS Layer

Application-facing analytical tables used by BI systems.

Example tables:

```
ads_trade_overview
ads_trade_top_category
ads_trade_top_shop
```

These tables power the BI dashboard.

---

# Partition Strategy

To support scalable analytical queries, warehouse tables are partitioned by date.

```
PARTITIONED BY (dt)
```

Example structure:

```
ads_trade_overview
 └── dt=2024-07-01
 └── dt=2024-07-02
 └── dt=2024-07-03
```

Benefits:

- efficient incremental loading
- reduced scan cost
- faster analytical queries
